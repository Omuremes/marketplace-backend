# Сессия 5: Аутентификация, JWT, Разделение ролей и Инфраструктура

## Цель
Реализация надежной системы регистрации и авторизации продавцов (Sellers) с помощью JWT-токенов (срок жизни и Refresh токены), выделение администраторов в отдельную таблицу БД, решение проблем с совместимостью модулей шифрования паролей и аудит архитектуры согласно ТЗ.

## Что было сделано

1. **Разделение ролей и База Данных:**
   * Создана отдельная сущность и DTO для `Admin`. 
   * Сущность `Seller` дополнена полями `email` и `password_hash`.
   * Сгенерирована и применена миграция базы `0002_add_auth_tables.py` через Alembic, которая создала таблицу `admins` и добавила новые колонки в `sellers`.
   * Обновлены UoW, репозитории и мапперы для работы с новыми сущностями.

2. **JWT-токены и безопасность:**
   * Внедрена система Access и Refresh токенов (`ACCESS_TOKEN_EXPIRE_MINUTES = 15`, `REFRESH_TOKEN_EXPIRE_MINUTES = 43200` или 30 дней).
   * Вынесена логика по генерации JWT и хешированию/проверке паролей в инфраструктурный слой `app/infrastructure/auth`.
   * Решена ошибка `AttributeError` от `passlib` переписав модуль хэширования на "чистый" `bcrypt` напрямую. Ограничение в 72 байта также корректно обрабатывается обрезкой `[:72]`.
   * Изменена зависимость безопасности в FastAPI: `OAuth2PasswordBearer` заменен на `HTTPBearer` (`app/api/deps.py`), чтобы убрать лишние атрибуты (например, `client_id`, `client_secret`) из Swagger UI и позволить использовать JSON вместо Form-Data в `POST /login` эндпоинтах.

3. **API-эндпоинты:**
   * В роутер `app/api/v1/public/sellers.py` добавлены хэндлеры `POST /register` и `POST /login`. Оба при успехе возвращают DTO пользователя, `access_token` и `refresh_token`.
   * Обновлен `app/api/v1/admin/auth.py` (`POST /login`), который теперь сверяется с таблицей `admins` вместо хардкод констант из `settings`.

4. **Сидирование (Test Data):**
   * Обновлен скрипт `seed_data.py`. Теперь тестовые Sellers создаются с корректными `email` и `password_hash` (по умолчанию пароль "password").
   * В `alembic/env.py` корректно зарегистрированы модули `AdminModel` и `ProductAttributeModel`, чтобы автоматическая генерация миграций в будущем не затерла эти таблицы.

5. **Аудит проекта на соответствие ТЗ:**
   * Проведен полный анализ всех систем (`docker-compose.yml`, архитектура, тесты). Backend успешно удовлетворяет 100% требований задания (Асинхронный Postgres, FastAPI, MinIO, миграции, CORS, пагинация). 
   * Раскомментирована конфигурация сервиса `frontend` в `docker-compose.yml` (п 7.1).
